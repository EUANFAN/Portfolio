07Challenges & Solutions

The five genuinely hard problems, and how I solved them

I wrote this section the same way I would explain it in an interview: what made each problem hard, how I broke it down, what design I landed on, and what kinds of follow-up questions a strong interviewer usually asks next.

01Unified schema: one project model for many campaign types

The hard part was not “save a JSON.” The hard part was designing one schema that could serve the editor, runtime, SSR flow, and many different business use cases.

Why this was hard
  • Some campaigns were long single-page scroll experiences, while others behaved more like multi-page flows.
  • Some widgets were atomic, while others were containers with internal layers and their own child widgets.
  • Some content was static, while some depended on remote data or runtime variables.
  • The same data model had to be editable, renderable, serializable, and safe to publish.

Schema example
const projectSchema = {
  name: 'Fall Campaign',
  componentPlat: 'h5',
  pages: [
    {
      id: 'page_1',
      width: 375,
      height: 1200,
      widgets: [
        {
          id: 'banner',
          type: 'Image',
          version: '1.0.0',
          layout: 'flow',
          data: { url: 'https://cdn/banner.png' }
        },
        {
          id: 'tabs',
          type: 'Container',
          layout: 'normal',
          hasLayers: true,
          layers: [
            { id: 'layer_1', widgets: [...] },
            { id: 'layer_2', widgets: [...] }
          ]
        }
      ]
    }
  ]
}
What I optimized for
  • Layered modeling. Project / Page / Widget / Layer made the editor and runtime think in the same shape.
  • Business data sits under widget.data. Platform fields and business fields stay cleanly separated.
  • Behavior is not mixed into style. Triggers, animation, and conditions stay independently consumable.
  • The schema carries more than UI. Variables, data sources, and platform config are part of the project model too.
Why not keep the schema flatter or lighter?
Because we were not building a static page renderer. We were building a full campaign project model that had to support editing, previewing, publishing, runtime execution, and third-party component integration.
Why put business fields under widget.data instead of the widget root?
The platform needs a stable set of fields it can understand universally, like layout and selection state. Business-specific fields vary by widget and should not pollute the platform contract.

02Editor core: three-pane collaboration and multi-level viewports

The real challenge was not laying out left, center, and right panels. It was making them behave like one editor instead of three disconnected features.

Why this was hard
  • The left side was not just a widget list. It also held pages and the layer tree.
  • The center area was not a single canvas. It had to move across page, container, and layer contexts.
  • The right side had to change based on both the current editing stage and the current selection.
  • Some widgets lived in flow layout, while others lived in freeform drag-and-resize layout.

MobX was a big part of why this stayed manageable. The editor was full of continuously changing object state: selected widget, current page, current layer, positions, sizes, tab state, viewport level. That fit MobX much better than a model based on lots of manual event passing.

Why we used multi-level viewports
Select a container widget → Switch current stage to widget → Pick a layer → Switch current stage to layer → Workspace moves into a new viewport
What was MobX doing here that really mattered?
It acted as the live synchronization layer for the editor. The three panes did not coordinate through hand-written UI sync logic. They all reacted to one observable project tree.
Why not Redux or an event bus?
The state was deeply nested and changed constantly. We wanted direct object edits with automatic downstream updates, not a lot of ceremony around dispatching and reducing tiny edits.

03Rendering convergence: making the editor and production runtime stay close

WYSIWYG is easy to say and hard to earn. The challenge was to make editing, preview, and runtime render through as much shared logic as possible.

Core idea
  • Both sides used the same schema and the same widget protocol.
  • Container layout logic was shared instead of rewritten separately.
  • Widget rendering converged on the same onRender path.
  • The editor worked on a fixed 375px canvas, while production translated the same values through px2rem.
What made it practical
Shared schema → Shared container logic → Shared widget protocol → Shared onRender output → Less drift between editor and production
Why not maintain separate rendering paths for the editor and runtime?
Because the drift would show up immediately. Separate renderers might move faster in the short term, but every visual bug would turn into a synchronization problem between two systems.

04Component platformization: widget protocol and dynamic loading

This stopped being a page builder and started becoming a platform once other teams could ship widgets without changing the platform core.

What the protocol needed to cover
  • type and version for stable lookup
  • data for business inputs
  • config for property panel generation
  • methods and listeners for runtime behavior
  • onRender as the actual rendering contract
Dynamic loading flow
Schema references widget type + version → Runtime fetches JS/CSS from CDN → Widget self-registers → Registry merges it with built-ins → Editor and runtime can render it
Why not just publish everything as npm packages?
Because we needed runtime version pinning per campaign page, and we needed historical pages to stay stable without forcing a platform rebuild. CDN-based versioned bundles fit that model much better.
Why was config part of the widget contract?
Because a widget needed to be editable, not just renderable. The editor had to know how to generate forms for its properties without hardcoding every widget type into the platform.

05Publish pipeline: SSR pre-rendering with graceful fallback

This was the most failure-prone path in the whole system, because it had to combine dynamic widgets, server rendering, CDN publishing, and runtime recovery.

Publish flow
1. Prepare
Clone revision dataBuild publish-ready PageData
Boot jsdomCreate a browser-like environment in Node
2. Load
Resolve used widgetsFind the actual component versions in the schema

3. Render
renderToStringProduce first-screen HTML

4. Deliver
Upload to CDNShip HTML, JSON, and static assets
Fallback if neededClient render remains the safety net
What made it stable
  • If SSR preload failed, the publish flow still continued.
  • If the React render itself failed, the page still published with client-side rendering available.
  • The runtime chose between hydrate and render based on whether SSR was truly usable.
  • We cleared require.cache between publish runs to avoid stale module state contaminating the next render.
Why not block publishing when SSR fails?
Because the business SLA was “the page must ship.” SSR improved first paint, but the publish pipeline itself mattered more. We treated client rendering as a real fallback, not an afterthought.
Why not use Next.js here?
This was not a normal app-router SSR problem. It was a schema-driven publish service with dynamic widget bundles and versioned component injection. A custom Koa + jsdom path gave us much more direct control.
