项目深度梳理

企业级低代码活动平台

一个面向运营、产品、市场团队的可视化搭建平台。非研发同学通过拖拽组件、配置参数就能产出一张完整的营销活动页， 支持多端（移动 Web / 微信 H5 / 微信小程序）一次配置、同步发布，把活动上线周期从「几天」压缩到「十几分钟」。

01项目概览

一句话讲清楚这个项目

一个 JSON schema 驱动的低代码活动搭建平台：左侧物料库 + 中间画布 + 右侧属性面板是运营看到的编辑器； 底层是一套自研渲染引擎 + 版本化组件库 + 触发器事件系统，让同一份 schema 在多个渠道上都能正确渲染、正确埋点、正确分享。

背景（S）
早期每个活动页都由前端手写，运营一有「换图、换文案、加个按钮」的需求就得排期。团队里一半的前端人力被消耗在重复的落地页搭建上，上线节奏被研发工期卡死。
目标（T）
把前端从活动交付链路中「拿掉」：运营自己配、自己预览、自己发布；同时必须保证多端一致、视觉不走样、数据埋点不丢。
动作（A）
设计一套能完整描述页面结构、样式、行为、分享配置的 JSON schema；构建渲染引擎 + CDN 版本化组件库 + 拖拽式编辑器；带动 4 条业务线完成迁移。
结果（R）
活动页上线周期从平均 2 天降到约 15 分钟；前端团队从「交付型」转向「平台型」工作——维护物料库、优化编辑器体验、接入实验与埋点能力。

让它真正跑起来的几个关键点

  • Schema 即契约。一份 JSON 描述布局、组件、样式、触发器、分享配置。编辑器只负责写它，运行时只负责读它，两边之间没有任何其他耦合。
  • 组件版本化 + CDN 加载。每个 widget 是一个独立 bundle，有自己的 semver；运行时按 schema 里钉住的版本加载，组件升级不会打穿历史活动页。
  • 用触发器代替「写代码」。不让运营写 JS，而是暴露一组声明式触发器（onClick/onShow/onScroll），它们的 action 也是 schema——跳转、埋点、分享、弹窗、跨组件调用。
  • 多端天然一致。同一份 schema 通过渠道适配器分别渲染到 H5、微信 H5、小程序——运营「配一次，发多端」。
