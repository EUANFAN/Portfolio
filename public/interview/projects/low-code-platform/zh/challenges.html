01MobX 动态属性追踪陷阱

一个组件的属性值改了，但右侧面板刷不出来——这个 bug 让我们踩穿了 MobX 响应式的边界。

问题场景

项目早期我们踩过一个很诡异的 bug：运营在右侧属性面板里给图片组件填了一个新的 alt 字段， MobX 没有触发画布重绘。但再填已有的 url 字段，就正常更新。 明明用的是同一个 observable，为什么新字段不生效？

这背后是 MobX 4 的核心限制：

  • observable({}) 只会把创建时已有的属性包装成 observable。
  • 后来动态 widget.data.alt = 'xxx' 加的新属性，不在 MobX 的追踪图里，不会触发 reaction。
  • 但"编辑器里组件属性会被动态加减"恰恰是我们的核心场景——每个组件的 config 决定它有多少字段，字段是动态的。

这个问题如果不解决，整个编辑器的响应式就是间歇性失灵的，运营用两下就会抱怨「这破东西又不更新」。

功能实现架构图
问题根源
observable({}) 只追踪创建时的属性
widget.data.foo = 'x' 新属性 → 不触发 reaction
解决方案（Widget 基类兜底）
modifyData(key, value) 整块重赋值触发引用变更
this.data = { ...this.data, [key]: value } 强制 MobX 检测到变化
使用侧（业务代码）
属性面板 onChange 调 widget.modifyData('alt', v)
其他字段也走这条路径 统一且无脑
结果
@observer 自动重绘 任何字段都触发
业务代码无感知 不用管 MobX 细节
核心思路

既然 MobX 4 追踪的是对象引用而不是对象内部的动态字段，那我就每次修改都换一个新引用。 把这件事放进 Widget 基类的 modifyData 方法里，业务代码永远只调这个方法， MobX 的坑就被藏在基类里，不渗透到业务层。

代码对比
Before · 直接赋值（间歇性失灵）
// 业务代码：右侧属性面板 onChange
const handleAltChange = (v) => {
  widget.data.alt = v   // ❌ 动态属性，MobX 4 不追踪，画布不会重绘
}
After · 基类兜底 + 整块重赋值
// store/clazz/Widget.js —— MobX 基类
class Widget extends Base {
  @observable data = {}

  modifyData(key, value) {
    // 关键：整块重赋值，引用变化 → MobX 感知 → 下游重绘
    this.data = { ...this.data, [key]: value }
  }
}

// 业务代码：永远走 modifyData
const handleAltChange = (v) => {
  widget.modifyData('alt', v)   // ✅ 任何字段都能触发更新
}
关键细节（面试官可能追问的点）
  • 为什么不升级 MobX 到 5+？MobX 5 用 Proxy 解决了动态属性追踪，但 ① MobX 5 不兼容 IE11，我们当时还要支持 App 内嵌 WebView（部分 Android 版本卡在老 Chrome）； ② 升级成本大，整个代码库的装饰器写法要改； ③ 已有兜底方案成本更低。所以主动选择"在 MobX 4 的边界内解决问题"。
  • 为什么不把 widget.data 声明成 observable.map？ observable.map 追踪 key 的动态增减，也能解决。但 map 的 API（data.get(key) / data.set(key, v)）和普通对象不兼容—— 组件的 onRender(ctx) 里到处都是 ctx.data.url 这种点语法，换 map 要全改。兼容现有使用姿势更重要。
  • 批量改字段会不会性能差？每次都重建 data 对象，看起来会有开销。实测下来一个组件的 data 通常不超过 20 个字段，浅拷贝成本忽略不计。 如果真的在热路径调用，可以包 runInAction 批量合并，但我们没遇到这种场景。
  • 撤销 / 重做怎么配合？撤销栈记录 data 的快照（浅拷贝）。Undo 时直接 this.data = snapshot，也走引用替换——MobX 自动触发重绘，无需额外处理。
  • 基类强约束怎么落地？ESLint 自定义规则，禁止业务代码直接写 widget.data.xxx = 。扫到就报错。新人上手一次就记住。

面试这样说

"项目早期我们遇到一个特别难排查的 bug——右侧属性面板改了组件字段，画布不更新。查了一圈才发现是 MobX 4 的一个已知限制：observable({}) 不追踪 后来动态加的属性。但编辑器里"字段是动态的"恰恰是核心场景，这个限制不兜底，整个响应式就是间歇性失灵的。"

"我的解法是在 Widget 基类里加一个 modifyData(key, value) 方法，内部永远用 this.data = { ...this.data, [key]: value } 整块重赋值，靠引用变化让 MobX 感知到更新。 业务代码从此只调这个方法，不用理解 MobX 的内部机制。"

"我没有选择升级 MobX 5 或者换 observable.map，是因为升级成本和兼容性代价都比在 MobX 4 边界内解决大得多。 这种"把框架局限收敛在基类里"的做法，也是我后来写平台型代码一贯的风格。"

MobX 5 / 6 完全解决了这个问题，为什么不直接升级？
三个原因：① 兼容性——MobX 5 依赖 Proxy，不支持 IE11 和部分老 Android WebView；② 代码迁移成本——装饰器语法和 observable API 有变化，几千行代码要挨个改；③ 我们的场景里组件的 data 字段数量有限（20 个以内）， 整块重赋值的开销基本为零。"能用现有工具解决的问题，不升级"是我的一贯判断。
如果 data 里嵌套了对象（比如 data.style.color），这套兜底还生效吗？
生效需要嵌套对象也走同样的"重赋值"模式：widget.modifyData('style', { ...widget.data.style, color: 'red' })。 我们把这个递归写进了一个 modifyDeepData(path, value) 工具函数，业务层统一调它。 更深的嵌套在我们的场景里不多，用 immer 也可以，但会引入 immer 依赖，当时没必要。
性能上，每次改字段都做浅拷贝，会有瓶颈吗？
理论上会，但实测没到瓶颈。原因：① 一个 widget.data 通常不超过 20 个 key，浅拷贝 cost 忽略； ② 属性面板 onChange 的频率不会高到压垮主线程（人手输入而已）； ③ 真的在热路径（比如滑块连续拖动），我们用 debounce 或 runInAction 批量合并，也能压住。
为什么不直接用 Redux，这些坑不就自然不存在了？
Redux 确实没有动态属性追踪这个问题，但编辑器的状态嵌套深度和改动频率用 Redux 要写巨量的 action / reducer。 我们权衡过——MobX 虽然有动态属性的坑，但只需要一个基类方法就能兜底；Redux 虽然没这个坑，但要付出的样板代码成本是持续的、永久的。 一次性的代价 vs. 持续性的代价，我选一次性的。

02所见即所得的双端一致性

编辑器里搭得好好的，一上线跑偏——这是低代码项目最怕的事，我们靠 5 层策略压住。

问题场景

低代码平台最致命的问题是"编辑器里看的是一个样，线上跑出来是另一个样"。 运营 debug 一次要来回"编辑 → 发布 → 打开 C 端看 → 回去改 → 再发布"，一次 10 分钟，一天能耗 2 小时。

但编辑器和 C 端天然有差异：

  • 编辑器是 React Class + MobX，C 端是 React 客户端 bundle + SSR hydrate。
  • 编辑器要交互层（拖拽 / 缩放 / 选中高亮 / 辅助线），C 端要触摸层（点击 / 滑动 / 懒加载）。
  • 编辑器画布宽度固定 375px，C 端要按视口宽度自适应。

如果两端各写一份"渲染图片组件"的代码，样式细节不可能完全对齐，迟早出问题。

画布宽度策略

C 端 · px2rem（1rem = 37.5px） 视口等比缩放
核心思路

把两端"真正相同"的部分强制收敛到一处，把两端"合理不同"的部分明确隔离。 具体就是 5 层策略：共享基类、共享渲染函数、共享样式文件、共享布局逻辑、px2rem 自动对齐。 组件开发者只写一份 onRender 和一份 .less，剩下的事让基类接住。

代码对比
Before · 编辑器和 C 端各写一份（视觉差异频发）
// client/widgets/Image/EditorImage.js —— 编辑器版本
class EditorImage extends Component {
  render() {
    const { url, width, height } = this.props
    return <img src={url} style={{ width, height }} />
  }
}

// client/widgets/Image/ClientImage.js —— C 端版本
class ClientImage extends Component {
  render() {
    const { url, width, height, alt } = this.props
    // ⚠️ 多了 alt 处理，少了 fallback，两端渲染结果不一致
    return <img src={url} alt={alt} style={{ width: width + 'rem', height: height + 'rem' }} />
  }
}
After · 共享基类 + 共享 onRender + 环境适配
// client/widgets/WidgetClass.js —— 双端共享基类
class WidgetClass {
  getStyle() {
    const { width, height, left, top } = this.props.widget.data
    return { width, height, left, top }   // 样式计算在一处
  }

  getWidgetContent() {
    const { widget } = this.props
    const ctx = { data: widget.data, methods: widget.methods }
    return widget.onRender(ctx)   // 渲染函数在一处
  }
}

// client/widgets/Image/index.js —— 组件定义（唯一一份）
export const Image = {
  type: 'Image',
  onRender(ctx) {
    const { url, alt, fallback } = ctx.data
    return <img src={url || fallback} alt={alt} />
  },
}

// 编辑器侧：加交互包装
class BaseWidget extends WidgetClass {
  render() {
    return (
      <Rnd style={this.getStyle()}>
        {this.getWidgetContent()}   // ← 共享渲染
      </Rnd>
    )
  }
}

// C 端侧：加触摸 + 懒加载 + px2rem
class Widget extends WidgetClass {
  getStyle() {
    return px2rem(super.getStyle())   // ← 覆盖 getStyle 加 px2rem
  }
  render() {
    return (
      <LazyLoad>
        <div style={this.getStyle()} onClick={this.triggerClick}>
          {this.getWidgetContent()}   // ← 共享渲染
        </div>
      </LazyLoad>
    )
  }
}
关键细节
  • onRender 是"纯函数"级别的契约。入参是 ctx（data + methods），出参是 React 元素。 不允许访问 DOM、不允许用 window——因为 SSR 时 window 不存在。这条写进规范里，组件评审卡住。
  • 样式文件只有一份 .less。编辑器和 C 端都 import 它。这是"视觉一致"最硬的一刀——差异没有机会出现。
  • px2rem 在 C 端的 getStyle 里做。编辑器画布定宽 375，像素直接用；C 端按视口动态算 rem。 逻辑只在 C 端的 Widget.getStyle() 里，组件开发者完全无感。
  • 布局逻辑（flow + normal）也共享。Container.js 负责把子组件按 "normal"（自由定位）和 "flow"（文档流）两类分别渲染——两端都调用它。 编辑器拖进去是什么样，C 端就是什么样。
  • 交互层严格隔离。编辑器的 Rnd / 选中框 / 辅助线绝不出现在 WidgetClass 里； C 端的 LazyLoad / 触摸事件也绝不出现在共享层里。基类只负责"内容和样式"，包装层各自加交互。

面试这样说

"低代码平台最致命的问题是编辑器和 C 端渲染不一致——运营调好一个活动页，发布上线跑偏了，得来回反复调整。 我们遇到的真实 bug 基本都是"编辑器有个效果，C 端没有"，因为两端的渲染代码各写了一份，细节很难对齐。"

"我的解法是把"真正相同"的东西强制收敛——做了个 WidgetClass 基类，编辑器的 BaseWidget 和 C 端的 Widget 都继承它； 组件开发者写一份 onRender(ctx)、一份 .less，两端共用。编辑器加 Rnd / 选中框等交互层，C 端加触摸 / 懒加载 / px2rem—— 差异部分明确隔离在各自的包装里。"

"画布宽度也做了对齐——编辑器固定 375px，C 端 px2rem 等比换算（1rem = 37.5px）。这样运营在编辑器里看到的像素尺寸， 就是 iPhone 上线后的真实尺寸，其他宽度按比例缩放，视觉永远不跑偏。"

"这套做法上线后，视觉不一致的 bug 从每周 3-5 个降到每月不到 1 个，组件接入时间也从 2 天降到半天。"

为什么选 375 作为画布基准宽度？
iPhone 的基准宽度就是 375px，设计稿 99% 是按 iPhone 出的。选 375 意味着运营看到的画布和设计稿 1:1 对齐， 视觉上心智最小。C 端用 px2rem 等比缩放到实际视口宽度，大多数手机宽度在 320-428 之间，缩放比例在 0.85-1.14 之间，失真很小。 iPad（768+）确实会拉得太大，但我们业务投放 99% 在手机上，这个代价可以接受。
px2rem 的转换是在构建时还是运行时？
我们选的是运行时——C 端 Widget.getStyle() 里通过工具函数转换。原因是活动页的宽度、位置来自 schema（动态数据）， 构建时不知道具体数值。另一种做法是用 postcss 插件在构建时把静态 CSS 里的 px 转 rem，但我们的组件库样式文件走这条， schema 里的动态 style 走运行时，两套并存。
如果某个组件在编辑器里有特殊的视觉（比如拖拽时的蓝色边框），这个怎么处理？
这种编辑器专属视觉放进 BaseWidget 里，不进共享层。原则很清楚： 组件内容和业务样式是共享的，编辑器 UI 装饰是专属的。蓝色边框、选中高亮、辅助线、标尺线——都属于编辑器 UI 装饰。 组件开发者碰不到这些，也不应该碰。
onRender 里禁止用 window，SSR 时是怎么兜底的？
两层保护：① 规范——组件评审时会检查 onRender 代码是否引用了 window / document，禁止通过； ② 运行时兜底——SSR 的 jsdom 环境里给 window / document 做了最小化 mock，就算组件误用也不会整个 SSR 挂掉； ③ 真挂了就走优雅降级，needHydrate = false，客户端兜底渲染。

03可插拔组件体系 + 运行时加载

怎么让第三方同学发一个组件不用等平台发版、但又不能打穿历史活动页。

问题场景

业务组件有三个截然不同的需求，塞进同一种加载方式都不对：

  • 全部打包进客户端？ 那第三方团队发一个"活动专属倒计时"组件要等平台发版，一周起步； 客户端 bundle 会随组件数量线性膨胀，几百个组件后首屏崩。
  • 全部运行时从 CDN 拉？ 那 Image、Container 这种基础组件都要等 CDN 回来才能渲染，首屏体验差； 而且 SSR 里要先串行 fetch 一堆 JS，发布速度也慢。
  • 组件升级怎么办？ 新版本的 Image 组件改了默认样式——会不会把半年前发布的活动页打穿？这是历史活动页稳定性的核心挑战。
功能实现架构图
层 1 · 内建组件（打包进客户端）
Image · RichText · NormalText 所有业务都会用
Container · DataContainer · Layer 底层结构组件
构建时 import 首屏秒出
层 2 · 内部组件（PageData.innerWidgets · CDN）
CourseCard · 报名表单 · 老师介绍 业务定制 · 稳定使用
loadScript + loadStyle 动态加载 首次使用时拉取
层 3 · 外部组件（PageData.installed · x-cli）
第三方团队 x-cli 开发 独立 bundle + xcli 元数据
CDN: /components/{type}/{version}/index.js 版本化存储
schema 钉死 version 老活动永远用老版本
统一注册中心
WidgetConfigs（@observable） 不管哪一层，最终都走这里
god.__components__ 外部组件自注册入口
核心思路

按"稳定度和使用频率"把组件分三层：最稳定最高频的打进客户端（内建），比较稳定但业务化的 CDN 动态加载（内部）， 第三方的完全独立发布（外部）。三层共用一套组件契约（onRender / data / config）， 最终都汇聚到 WidgetConfigs 注册表。版本钉在 schema 里——老活动用老版本、新活动可升级，互不干扰。

Before · 如果全部静态 import

After · 三层分级 + 动态注册
// client/widgets/index.js —— 内建组件静态注册
import Image from './Image'
import RichText from './RichText'
import Container from './Container'
// ... 只有基础组件

export const WidgetConfigs = observable([Image, RichText, Container, ...])

// 运行时：从 schema 读需要的外部组件，动态加载
export async loadExternalWidgets(pageData) {
  const tasks = pageData.installed.map(({ type, version }) => {
    const jsUrl  = `${STATIC_URL}/components/${type}/${version}/index.js`
    const cssUrl = `${STATIC_URL}/components/${type}/${version}/index.css`
    return Promise.all([loadScript(jsUrl), loadStyle(cssUrl)])
  })
  await Promise.all(tasks)
  // 组件 JS 执行后会自注册到 god.__components__
  const externalConfigs = Object.values(god.__components__)
  WidgetConfigs.push(...externalConfigs)   // MobX 数组 push 触发下游
}

// 外部组件开发方（用 x-cli 脚手架）
// src/index.js —— 编译后最终注册到 god
registerComponent({
  type: 'Countdown',
  xcli: { version: '1.2.0', ssr: true },   // xcli 元数据声明 SSR 兼容性
  data: { endTime: '', format: 'HH:mm:ss' },
  config: { endTime: { type: 'DateTimePicker' }, format: { type: 'Input' } },
  onRender(ctx) { /* ... */ },
})
关键细节
  • 版本钉死在 schema 里。发布时把当前使用的组件 version 写进 pageData.installed。 之后组件迭代到 1.3、1.4，历史活动的 schema 仍然写着 1.2.0，C 端按 schema 拉 1.2.0 的 JS——永远不会被升级打穿。
  • 想升级就在编辑器里显式操作。编辑器提供"检查组件更新"入口，运营主动点"升级到 1.4"，schema 才会改。 默认不动 = 默认稳定，升级是运营的一次"确认"。
  • SSR 能力由组件声明。只有 xcli.ssr === true 的外部组件才会在服务端执行，避免第三方组件里用了 window 导致 SSR 崩溃。 不兼容 SSR 的组件自动走 hydrate-only 模式。
  • god.__components__ 是外部组件注册入口。组件 bundle 最后一行调用 registerComponent(config)， 内部往 window.god.__components__[type] = config 里塞。平台再从 god.__components__ 收集、push 到 WidgetConfigs。
  • loadScript 防重复。同一个 JS URL 只加载一次，重复调用直接 resolve。用一个 Map 缓存 Promise。
  • x-cli 脚手架统一规范。第三方同学用脚手架初始化一个组件项目，写完 npm run build 产出符合规范的 bundle + 元数据文件。 开发者不用关心"什么是 god.__components__"——脚手架自动包好。

如果 1.2.0 的组件 JS 在 CDN 上被误删了怎么办？
两层兜底：① CDN 层——组件 JS 走版本化目录（/components/Countdown/1.2.0/index.js），删除需要人工操作 + 审批； ② 运行时——loadScript 失败时，对应的组件会渲染成一个占位元素（「组件加载失败」），不影响页面其他部分。 SSR 侧如果拉不到就走降级（needHydrate=false），让客户端自己去 fetch，客户端再失败就是占位。层层兜底。
怎么防止第三方组件在页面里搞破坏（比如改 window 全局）？
说实话我们没做严格的沙箱。一是 iframe / Web Worker 隔离成本太高，组件间通信复杂； 二是第三方开发者来自公司内部各业务团队，不是完全陌生的来源，审核流程可以解决大部分风险。 但我们做了几件事：① x-cli 脚手架对代码有静态检查（禁止 window.xxx = 、禁止修改 god）； ② 代码评审必须通过平台组；③ 线上运行时监控，发现异常组件可以一键冻结。"审核代替沙箱"是权衡后的选择，对内部生态够用。
外部组件要配合 SSR，肯定有一堆坑，怎么管？
分两层：① 声明制——组件的 xcli.ssr 字段明确声明是否支持 SSR。不支持就 hydrate-only，不会在服务端执行； ② 检测工具——x-cli 脚手架提供 xcli ssr-check，扫代码里是否引用了 window / document / localStorage，有就警告； ③ 运行时兜底——jsdom 做了最小化 mock，真挂了还有优雅降级。
组件版本升级时，如果新老版本的 data 字段不兼容（比如老版本有 color，新版本改名 themeColor）怎么办？
组件升级遵循 semver——大版本升级才能做字段 breaking change。升级流程： ① 组件开发者在 x-cli 里声明 migration 函数（从 1.x 的 data 转换到 2.x 的 data）； ② 运营在编辑器里点"升级到 2.0"时，平台自动调 migration 转换 schema； ③ 如果没写 migration，平台不让升级，只能新组件 + 老组件并存。这样把"破坏性变更"的责任交给组件开发者，平台只做协议。

04SSR 发布链路 + 优雅降级

要让 Node 模拟浏览器跑 React，还要保证任何一步失败都不阻塞发布——这条链路是整个项目最密的地方。

问题场景

C 端首屏 LCP 要在 1.5s 以内——不用 SSR 根本达不到。但 SSR 要在 Node 侧跑 React + 第三方组件代码， 问题一下子来了：

  • Node 没有浏览器 API。组件一调 document / localStorage / navigator 就炸。
  • 第三方组件是动态的。不同活动页用不同组件、不同版本——SSR 前必须先从 CDN 把 JS 拉下来执行注册。
  • Node require 有缓存。上次发布加载的 App 模块和组件会留在 require.cache，下次发布直接复用——组件版本错乱。
  • SSR 很容易失败。组件 JS 下载超时、SSR 里误用 window、第三方组件抛错……不能因为 SSR 挂了就整个发布失败。
功能实现架构图
Step 1 · 环境准备（jsdom）

god 全局挂载 组件注册入口
require.cache 清理 防上次发布污染
Step 2 · 组件预加载

xcli 元数据校验 只允许标记 ssr:true 的组件执行
Step 3 · React SSR
require 客户端 App 拿到 React 树入口

Step 4 · 发布
HTML / JSON 上传 CDN 统一 CDN 路径
自动截图生成缩略图 编辑器预览用
Step 5 · 失败降级
任何一步挂掉 needHydrate = false
C 端退回 ReactDOM.render 首屏慢一点，但绝不白屏
核心思路

SSR 必须在 Node 侧复刻一个足够接近浏览器的运行环境，同时每一步都要有失败兜底。 关键是三件事：① jsdom 构造假浏览器；② 按 schema 动态预加载组件；③ 一层层 try/catch 保证任何失败都降级到客户端渲染，发布流程不中断。

代码对比
Before · 直接 renderToString（各种炸）
// server/controller/project/helper/index.js —— 不够健壮的早期版本
const App = require('../../client/App')
const html = renderToString(React.createElement(App, { pageData }))

After · 完整流水线（jsdom + 预加载 + 降级）
// server/controller/project/helper/index.js
const { JSDOM } = require('jsdom')
const { getComponentScript } = require('../../utils/script')

async renderPage(pageData) {
  let mainHTML = ''
  let needHydrate = true

  try {
    // 1. 构造 jsdom 假浏览器
    const dom = new JSDOM('<!DOCTYPE html>', { url: 'https://example.com' })
    global.window = dom.window
    global.document = dom.window.document
    global.navigator = dom.window.navigator
    global.localStorage = createMockStorage()
    global.god = { __components__: {} }

    // 2. 清 require.cache 防污染
    clearAppCache()

    // 3. 按 schema 拉外部组件 JS，new Function 执行注册
    for (const { type, version, xcli } of pageData.installed) {
      if (!xcli?.ssr) continue   // 不支持 SSR 的跳过
      const code = await getComponentScript(type, version)
      new Function('god', 'window', code)(global.god, global.window)
    }

    // 4. require 客户端 App，调 renderToString
    const App = require('../../client/App')
    mainHTML = renderToString(React.createElement(App, { pageData }))
  } catch (err) {
    // 5. 任何一步失败都降级，不阻塞发布
    log.warn('SSR failed, fallback to client render', err)
    mainHTML = ''
    needHydrate = false
  } finally {
    cleanupGlobals()   // 把 window / document 从 global 删掉
  }

  // 6. Handlebars 装配 HTML + PageData
  const finalHTML = template({ mainHTML, pageData, needHydrate })
  return finalHTML
}
关键细节
  • require.cache 必须清。Node 的 require 会缓存模块。上次发布加载了 Countdown 1.2，下次发布 1.4，如果不清缓存就还是 1.2。 清理策略是：清 App 入口及其所有依赖（递归清 require.cache[key]），不清 node_modules——那些是稳定的库。
  • jsdom 是有状态的。每次 SSR 都 new JSDOM()，用完在 finally 里把 global.window 等删掉。 否则并发 SSR 两个活动页，两边互相污染——这个坑我们排查了整整 2 天。
  • xcli 元数据校验。只有组件 bundle 里带 xcli.version 且 xcli.ssr === true 的才在服务端执行—— 防第三方组件里用了浏览器 API 整个炸。没标 SSR 的就在客户端 hydrate 时再渲染。
  • new Function() 而不是 eval。new Function 有独立作用域，不会污染外层变量。参数列表里把 god / window 显式传进去—— 组件代码执行时这两个就是它能拿到的上下文。
  • 优雅降级是硬指标。无论 SSR 出什么幺蛾子，finalHTML 一定要返回。只是 mainHTML 是空的、needHydrate=false 而已—— 客户端拿到空壳 HTML 后 ReactDOM.render（不是 hydrate），照常渲染。
  • 截图生成也放发布链路里。HTML 传 CDN 后用 puppeteer 访问一次，生成缩略图——编辑器列表页展示用。这个失败不影响主流程。

"整条链路分五步：① jsdom 在 Node 里构造假浏览器； ② 按 schema 拉外部组件 JS，用 new Function() 执行，让它们注册到 god.__components__； ③ renderToString(<App />) 生成 HTML； ④ Handlebars 装配进模板传 CDN； ⑤ 任何一步挂掉，needHydrate=false，客户端退回 render 兜底。"

"上线后 LCP 从 3.2s 降到 1.4s，发布成功率靠优雅降级保住 99.9%。"

为什么不用 Next.js 或者其他成熟的 SSR 框架？
Next.js 适合"一个应用一个站点"的场景，但我们是"一个服务给几百上千个活动页打 SSR"，每个活动的组件、版本、schema 都不一样。 把这种"多租户 + 组件动态注入"塞进 Next.js 会非常别扭：要么把每个活动页包装成一个 Next 路由（服务起不来），要么用 Next 的编程式 API 绕一大圈。 最后还不如自己基于 Koa + jsdom 直接控制每一步。"用现成方案就要接受它的假设"——当假设不符时，自己搭反而更直接。
并发 SSR 时 jsdom 互相污染具体是什么现象？
我们最初是全局 global.window = jsdom.window，发布 A 页面时挂上，B 页面同时发布也挂上——两个请求共享 global.window， 后挂的覆盖前挂的。现象是 A 页面的 SSR HTML 里出现了 B 页面的组件数据。修复方案：① 改用 async_hooks 的 AsyncLocalStorage 做请求级隔离；② 或者限制 SSR 进程串行处理（我们选了这个——单个发布 ~8s，业务方可接受）。
require.cache 清理会不会影响性能？每次发布都重新 require 整条依赖链。
会，但可控。策略是：① 只清 App 入口 + 组件相关的模块，node_modules 里的 React / MobX 等稳定库不清，复用； ② 用 Node.js 的 PM2 集群模式，单次发布的开销被多进程分担； ③ 发布本身是低频操作（每天几十到几百次），不是热点路径。实测每次额外开销 ~500ms，在总耗时的 5% 以内。
如果外部组件里有异步逻辑（比如 useEffect 里 fetch），SSR 怎么处理？
SSR 只渲染同步产出——useEffect 不会在服务端执行。所以 useEffect 里的 fetch 不会走 SSR，走 hydrate 后客户端执行。 对性能的影响是：这部分数据首屏看不到，hydrate 后才出。我们的建议是"首屏关键数据走 schema，次要数据走 useEffect"—— 把 fetch 放进 schema 层面的 DataContainer，SSR 时可以把数据一起算出来。
SSR 失败降级后，用户体验差多少？有没有考虑过预渲染快照？
降级后 LCP 从 1.4s 退回到 3s 左右（差 ~1.5s）——确实有感知，但没白屏。我们做了两个兜底： ① 降级率监控，超过阈值告警；② 预渲染快照——上一次成功的 SSR HTML 会缓存在 CDN， 如果这次 SSR 失败，直接复用上一次的 HTML（stale-while-revalidate 思路）。但 schema 变了快照就不能用，我们只在"微调场景"启用（比如只改了文案）。

05Stage 抽象层 + 多层视口

Project / Page / Layer / Widget 四种容器，如果散着写 if/else，代码里全是分支地雷。

问题场景

编辑器里有四种嵌套容器：Project（一个活动项目）→ Page（多个页面）→ Layer（图层/容器组件）→ Widget（叶子组件）。 每个容器都要能：加子节点、删子节点、选中子节点、排序子节点。

如果按容器类型写 if/else，代码会变成这样：

  • if (parent.type === 'Project') parent.pages.push(child)
  • else if (parent.type === 'Page') parent.widgets.push(child)
  • else if (parent.type === 'Layer') parent.widgets.push(child)
  • else if (parent.type === 'Widget') parent.layers.push(child)

每一个操作都要这样 4 条分支。编辑器里光"加节点"场景就有七八个（拖入、复制、粘贴、撤销、模板应用、跨层移动……），累计下来是几十处 if/else 地雷。 加一种新容器？所有地雷一起挪。

功能实现架构图
散写时的代码分布
拖入组件 if/else × 4
复制粘贴 if/else × 4
撤销重做 if/else × 4
... 几十个操作场景 每处都要 × 4

Stage(component, type) 动态绑定到正确的子节点数组
addChild() · removeChildren() 对四种容器都适用
selectChildren() · sortChildren() 跨层级统一

stage.addChild(child) 不关心容器类型
stage.removeChildren([a, b]) 批量也是这个 API
多层视口（SwipeableViews）
Project → Page → Layer 逐级深入编辑
stages 栈 push / pop 维护导航
核心思路

把四种容器的"子节点数组在哪"这个细节藏进 Stage 类的构造函数里—— 传进来什么容器对象，this.list 就自动绑到正确的数组（pages / widgets / layers）。 之后所有操作都针对 this.list，业务代码里永远只看到 Stage 这一个抽象，不用知道底下是什么容器。

代码对比
Before · 散写的 if/else 地雷
// components/HEStage/index.js —— 拖入新组件
onDrop(parent, child) {
  if (parent.type === 'Project') parent.pages.push(child)
  else if (parent.type === 'Page')    parent.widgets.push(child)
  else if (parent.type === 'Layer')   parent.widgets.push(child)
  else if (parent.type === 'Widget')  parent.layers.push(child)
}

// components/HELayerList/index.js —— 删除图层

  if (parent.type === 'Project') parent.pages = parent.pages.filter(...)
  else if (parent.type === 'Page')    parent.widgets = parent.widgets.filter(...)

// store/stage/Stage.js —— 容器多态封装
class Stage {
  constructor(component, type) {
    this.component = component
    this.type = type
    // 根据容器形状自动绑定子节点数组
    this.list = component.pages || component.widgets || component.layers || []
  }

  addChild(child) {
    this.list.push(child)
  }
  removeChildren(children) {
    this.list.replace(this.list.filter(c => !children.includes(c)))
  }
  selectChildren(ids) {
    this.list.forEach(c => { c.isSelected = ids.includes(c.id) })
  }
  sortChildren(fromIndex, toIndex) {
    const [moved] = this.list.splice(fromIndex, 1)
    this.list.splice(toIndex, 0, moved)
  }
}

// 业务代码：一行干净的调用
onDrop(stage, child) {
  stage.addChild(child)   // ✅ 无视容器类型
}
onDelete(stage, children) {
  stage.removeChildren(children)   // ✅ 无视容器类型
}
关键细节
  • Stage 不持有状态，只是个代理。stage.list 直接指向真实容器的子节点数组（MobX observable）， 所有变更都反映到原对象上。Stage 自己不存数据——避免双写 / 同步的复杂度。
  • stages 栈维护编辑导航。运营从 Project 进到某个 Page，再进到某个 Layer 编辑——用一个 stages: Stage[] 栈记录路径。 返回上一级就 stages.pop()。多层级导航天然成栈结构，代码简单得不像话。
  • SwipeableViews 做视觉上的"多层"。运营点进一个 Layer，画布滑动到下一层——就像 iOS 的页面导航。 视觉上的深入和数据结构里的 stages 栈是1:1 对应的。
  • Stage 支持跨容器的移动。一个 Widget 从 Page A 的 Layer 1 拖到 Page B 的 Layer 2—— 先 stageA.removeChildren([w])，再 stageB.addChild(w)。两个 Stage 就是两个代理，业务代码不变。
  • 和历史栈（undo/redo）天然配合。每个 Stage 操作（addChild / removeChildren / sortChildren）都会被历史管理器记录为可逆操作。 撤销时反向调用——Stage 的接口越小，历史管理越简单。
实际效果

面试这样说

"编辑器里有四种嵌套容器：Project → Page → Layer → Widget。每种都要能加/删/选/排序子节点。 早期代码里到处是 if (parent.type === 'Project') {...} else if (parent.type === 'Page') {...} 这样的分支， 光一个编辑器累计几十处地雷，加一种新容器要挪一遍。"

"我的解法是抽一个 Stage 类——构造函数接收一个容器对象，自动识别它的子节点数组（pages / widgets / layers）， this.list 绑到那个数组上。之后 addChild / removeChildren / sortChildren 等操作都针对 this.list， 业务代码里永远只看到 Stage，不用知道底下是什么容器。"

"配合这个，多层级导航用 stages: Stage[] 栈实现——进入某一层就 push，退出就 pop，视觉上用 SwipeableViews 做页面滑动。 数据结构里的栈和视觉上的"深入"是 1:1 对应的，代码简单得不像话。"

"上线后业务代码里的 if/else 分支从约 80 处降到 0 处，新容器接入只要改 Stage 构造函数，撤销重做也因为接口统一而变简单。 这是我做过的"收敛抽象"里最舒服的一次。"

为什么不在每种容器里都写一个 getChildren 方法（多态），而要单独抽 Stage？
确实可以。我考虑过两种方案： ① 在 Project / Page / Layer / Widget 类上都实现 getChildren() / addChild() 方法（多态直接落到数据类上）； ② 外面再套一层 Stage 做代理。选 ② 的原因是容器类本身有独立的业务语义（Project 有 shareConfig、Page 有 variableStore 等）， 不想把编辑器侧的操作（选中 / 排序）混进去。数据类只关心数据，编辑操作交给 Stage——单一职责。
如果未来要加第五种容器（比如 Group 组件），Stage 的 constructor 会越写越长吗？
不会，因为设计上 this.list = component.pages || component.widgets || component.layers || [] 就是按"第一个非空数组"识别， 新容器只要用 children 或其他已有字段，Stage 自动识别。真要加新字段，也就 || 后加一项，一行改动。 这是"约定优于配置"在平台代码里的一个小例子。
MobX 的 observable 数组能直接调 splice / replace 吗？有性能问题吗？
可以，MobX 的 observable 数组重写了这些方法，会触发最小化的 reaction——只通知真正受影响的 observer。性能上比手动 setState([...arr]) 还好， 因为它能精确知道哪个索引变了。replace 是 MobX 数组特有的 API，一次性替换整个数组内容，用在"删除多个"场景最方便。
撤销重做怎么和 Stage 结合？
历史管理器 HistoryStore 监听 Stage 的每个操作，记录"操作类型 + 参数 + 反操作"。比如 addChild(x) 被记录为 { do: () => stage.addChild(x), undo: () => stage.removeChildren([x]) }。 因为 Stage 接口只有四个方法，每个都有明确的反操作，历史栈逻辑非常简单——接口越小，history 越干净。 这也是我们当初选"外层代理"而不是"数据类多态"的隐性收益。
