07Challenges & Solutions

Five tough problems, and how we worked through them

Each challenge includes: the problem setup, the architecture diagram, the solution breakdown, code comparison, key details, the result, an interview-ready spoken version, and likely Q&A. The goal is simple: once you finish one section, you should be able to explain that topic clearly on your own.

01CRUD abstraction

With 20+ list pages, how do you avoid rewriting filtering, pagination, and request logic from scratch on every page?

Problem setup

The whole admin had 20+ list pages: course projects, templates, components, coupons, course products, physical goods, promotion tools, channel QR codes, user purchase limits, course purchase limits, and more. The page skeleton was highly similar:

  • At the top, a filter area: a group of search fields such as name, status, date range, creator, plus Search and Reset buttons.
  • In the middle, a table: columns, pagination, sorting, and column visibility.
  • On the right, an operation column: view, edit, copy, delete, publish/unpublish, all shown or hidden by permission.
  • At the top or bottom, sometimes a bulk action area: after selecting rows, the user can run actions in bulk.

If every page implemented that on its own, it created three real problems:

  1. Code volume exploded: a hand-written list page could easily reach 400-500 lines, and around 80% of that was boilerplate.
  2. Interaction behavior drifted: page A reset filters when paging, page B did not; page A searched when you clicked the table header, page B only searched when you clicked a button. Small differences kept growing over time.
  3. Maintenance became painful: if we wanted one global change like “persist filter state to the URL,” we had to touch 20+ files.

Core idea

Pull out the few things every list page has to do anyway, and turn them into 3 hooks plus 3 components. From that point on, a business page only had to do two things: 1. define the columns and 2. provide the request function. Filter state, pagination state, request timing, “keep filters while paging,” “reset current on a new search,” loading state, and column persistence all came from the primitive layer.

Before · a hand-written coupon list page (main boilerplate from a ~450-line file)

Key details (the things an interviewer is likely to ask about)
  • The signature of useTableSearch: input was { searchFn, defaultParams, transform }, and output was { tableProps, searchFormProps, refresh, params }. tableProps went straight into an antd Table, and searchFormProps went straight into the search-bar component. The outer page did not even need to know what internal state the hook was using.
  • The paging reset policy was unified: Search → reset current to 1; paging → keep the filters; Reset → clear params and reset current to 1. Before that, every page wrote its own version. After that, all list pages behaved the same by default.
  • Request timing: requests were triggered when defaultParams changed or when the page manually called refresh(). Field changes did not auto-search by default, to avoid accidental requests. That was a deliberate default after discussion.
  • The persistence key for columns: the second argument in useTableSet(columns, 'coupon-list') became the localStorage key, so one list page would not overwrite another page’s column settings.
  • Permission filtering in the action column: TableOperationDropdown accepted an actions array, and each action could carry an authKey. Internally it used useAuthCheck(authKey) to filter out unauthorized items, so business code did not have to repeat button-permission checks by hand.

How I would say this in an interview

“In admin projects, list pages are where repetition gets out of control the fastest. We had 20+ list pages, and the structure was almost always the same: filters at the top, a table in the middle, an action column on the right, and sometimes bulk actions as well. If every page is written from scratch, one page quickly grows to four or five hundred lines, and interaction details start drifting apart.”

“My answer was to extract a CRUD primitive layer: two hooks, useTableSearch and useTableSet, plus three shared components, FormSearchGroup, FilterComponent, and TableOperationDropdown. From that point on, business pages only needed to define columns and searchFn.”

“In the end, a new list page dropped from half a day to about two hours, and the same page went from 450 lines down to 80. More importantly, we got behavior consistency: users got the same experience on every list page, and that value came directly from the abstraction, not just from saving lines of code.”

Why not just use Ant Design Pro's ProTable directly?
ProTable really was the starting point, and we were already using it. But ProTable’s API is relatively “big,” and search, pagination, and requests are tightly coupled inside the same component. Once the business needed finer control, like “keep filters while paging, but reset sorting on a fresh search” or “some fields need debounce, others do not,” it became easy to fight ProTable’s defaults. So we added a thin wrapper of our own hooks and components around it. Simple lists could still stay close to ProTable, while more complex lists used our abstraction. Both layers existed side by side.
How was useTableSearch implemented internally?
At the core, it used useState to manage four pieces of state: params, pagination, dataSource, and loading. It exposed three handlers, onSearch, onReset, and onChange, and used useEffect to trigger searchFn at the right times. Using ahooks useRequest would also have been perfectly fine. We hand-wrote this part because we wanted full control over request timing. The key point was not which library we used. The key point was defining the rule once and applying it across the whole system.
What if one list page has a non-standard need, like drag-sorting the whole table?
My rule is do not force non-standard behavior into a shared component. If a page needs drag sorting, that page should implement it on its own. Shared components should stay focused on what most list pages genuinely have in common. If you stretch a shared API to fit the odd 10%, the other 90% end up paying for it. Over-abstraction is one of the easiest traps in B-side projects, so restraint matters.
Why not always store filter and pagination state in the URL? Otherwise refresh loses it.
By default we did not, for two reasons: one, the URL gets long and ugly very quickly; two, admin pages are not often shared in the same way as public-facing pages. For the smaller number of pages that really needed it, like “send the filtered result to a teammate,” useTableSearch could enable params → URL sync through a switch. Simple by default, configurable for edge cases is my usual way of handling tradeoffs like this.
How did you handle column-level permissions, where not everyone is allowed to see a certain column?
There were two ways. 1. A column definition could carry an authKey, and useTableSet would filter out unauthorized columns during initialization through useAuthCheck. 2. A business page could also conditionally render columns directly with useAuthCheck. The first way was more unified, and the second was more flexible. We supported both.

02Fine-grained permission system

Three layers of permission: menus, routes, and buttons. With 10+ roles, scattered if/else checks would have been a disaster.

Problem setup

More than 10 roles were using this admin at the same time: operations, marketing, product, customer support, supervisors, administrators, and more. Each role saw a different combination of menus, pages, and buttons:

  • Menu level: a standard ops role should not see the “System Management” menu; a product role should not see the “User Portrait” module.
  • Route level: someone might be allowed into the “Coupon List” page, but still not be allowed into the “Coupon Configuration” details page.
  • Button level: on the same page, one role can only view, a supervisor can “approve,” and an administrator can “delete.”

If permission checks were scattered through business code, we would quickly end up with dozens of conditions like if (user.role === 'admin' || ...). Once the permission rule changed, we would have to patch many places and would almost certainly miss a button somewhere.

Core idea

“One source of data, one place to declare it, three places to consume it.” All permission data was fetched once during startup; all permission switches were declared once in access.ts; and the business layer had three consumption points: routes, pages, and buttons. They were independent from one another, but all of them read from the same access object. Business code never touched raw permission data directly; it only consumed named switches from access.

Key details
  • The type convention: in the backend permission nodes, type=0 and type=1 meant menus, and the other values, like type=2, meant function buttons. The frontend recursively split those in getInitialState and pushed them into either menus[] or functions[]. That split was a contract agreed with the backend.
  • initialState was only fetched once at startup; we did not build real-time subscription. Permission changes were not frequent enough to justify that, so refresh or re-login was enough. If we ever needed real-time updates, Umi’s setInitialState could refresh the whole permission set.
  • Frontend permission was not the security boundary. We were explicit with the backend team about that: frontend interception only improved the UI, by hiding inaccessible entry points and avoiding pointless requests. The real security boundary was still enforced independently on the backend APIs. Even if a user manually edited the URL and bypassed the frontend shell, the actual data request would still be rejected by the backend.
  • The cost of useAuthCheck was tiny: at the end of the day it was just Array.includes on a string array. Even with a thousand keys, the cost would still be microseconds. Performance was not a concern here.
  • New-module onboarding cost: add one node in routes/index.ts and one switch in access.ts, and the module joined the system-wide permission model. No extra framework code needed to be touched.

How I would say this in an interview

“The biggest risk in B-side admin permissions is scattering permission logic through business code. Once you start writing if (user.role === 'xxx') across dozens of pages, the moment the rule changes, you have to patch a whole bunch of places and you still end up missing a button somewhere.”

“So my answer was one source of data, one place to declare it, three places to consume it. Fetch the permission tree once at startup, split it by type, feed it into Umi Max’s initialState, then generate 50+ semantic switches in access.ts, like canSeePresentCoupon. Routes use those switches, pages use <Access>, and buttons use useAuthCheck.”

“The benefit is that when the permission rule changes, I only update one place, access.ts. No raw permission strings appear anywhere in business code. A new teammate can open access.ts and immediately understand what permissions the system has and what each one controls. And we were also explicit that frontend permissions only improve the UI. The real security boundary still belongs to the backend APIs.”

What if permission changes have to take effect immediately, like an admin removing a permission and the button should disappear right away?
The default solution here was “refresh or re-login to take effect.” If real-time updates had become a requirement, we could have: 1. added long-polling or WebSocket notifications for permission changes; 2. called Umi’s setInitialState when the notification arrived; 3. let access re-compute against the new initial state, so business code would automatically react through useAccess. We did not add that extra layer in this project because permission changes were not frequent enough.
How did you stop people from forgetting permission config when they added a new page?
We had two checks: 1. in routes/index.ts, code review required every new route node to carry an access field; 2. if a route path existed but the permission tree did not grant access, the fallback behavior was a 403 page. So even if a newcomer forgot the config, the worst outcome was a 403 for the user, not accidental access to a page they should not see.
Did 50+ switches in access.ts become hard to maintain?
In practice, no. The reason was simple: we grouped them by business domain and commented them clearly. Project Center, Gift Center, Tools, and so on each had their own section. Adding a new switch was basically copy one line and rename it. That is still much cheaper than raw permission strings being scattered through hundreds of business files.
What is the difference between useAuthCheck and the <Access> component? When do you use which?
<Access> comes from Umi Max and consumes boolean switches from the access object. It is usually for a whole page or a whole block of UI, often with a fallback. useAuthCheck was our own wrapper around the raw functions array, and it was mainly for buttons or operation items where we just needed a boolean for conditional rendering. So: coarse-grained permission → <Access>; fine-grained permission → useAuthCheck.
If permissions are fetched at startup, what about first-load performance? Does the page feel slow?
Umi Max’s getInitialState is a Promise, so before it resolves, the app renders a loading component, usually loading.tsx. In practice the permission call took around 200-400ms, so the user only felt a small loading flash, which was acceptable. If we wanted to optimize further, two directions were possible: 1. HTTP caching + ETag on the permission API; 2. write a localStorage cache after one successful fetch, then boot from cache first and refresh in the background. In this project, we only used the first direction.

03SSO login and a unified request flow

How do you get the token, where do you store it, and what happens when it expires? Business code should not have to care about any of that.

Problem setup

The company used SSO for single sign-on, and multiple internal systems shared the same login state. On the frontend, we had to solve at least four things:

  • Getting login state: when a user visits for the first time, or jumps back from another subsystem with a token in the URL.
  • Injecting the token: every request needs to carry the login credential.
  • Showing business errors consistently: the backend had conventions around fields like stat and code; if a request failed at the business layer, we needed one shared message pattern.
  • Handling expired login: on 401, the user should be logged out through SSO, and after signing back in, land on the original page through redirect_url.

Core idea

“Business code should only care about the happy path.” Everything related to auth and error handling was pulled into the two interceptors on our axios instance. A business call should feel almost like a local function call: const user = await getUserInfo(). It returns a typed UserInfo directly, without the page worrying about response nesting, error-message rules, or whether the token has expired.

Code example
axios instance + request / response interceptors
// utils/axios.ts
const instance = axios.create({
  baseURL: config.host,
  timeout: 15000,
  withCredentials: true,
  headers: process.env.NODE_ENV !== 'production' ? { isDebug: 1 } : {},
})

// —— 请求拦截器：注入 token
instance.interceptors.request.use((conf) => {
  conf.headers = { ...conf.headers, Authorization: getToken() }
  return conf
})

// —— 响应拦截器：剥壳 + 业务错误 + 401 跳 SSO
instance.interceptors.response.use(
  (response) => {
    // 401：token 过期
    if (response.data?.code === 401) {
      message.error('用户未登录')
      return response.data.data
    }
    // 业务错误（约定 stat=1 才算成功）
    if (response.data?.stat !== 1) {
      message.error(response.data?.msg || '内部错误')
      return Promise.reject(response.data)
    }
    // 成功：剥壳返回 data.data —— 业务代码直接拿业务数据
    return response.data.data
  },
  (error) => {
    if (error?.response?.status === 401) {
      // token 过期：清 token + 跳 SSO logout，带 redirect_url 回跳
      message.error({
        content: '登录已过期，请重新登录',
        onClose: () => {
          postLogout().then((res) => {
            clearToken()
            window.location.href = `${ssoLogout}?path=${encodeURIComponent(res.redirectUrl)}`
          })
        },
      })
    } else {
      message.error('系统异常')
      return Promise.reject(error)
    }
  },
)

Key details
  • Token in cookie, not localStorage: a cookie can be written cross-domain by the SSO service, with the right domain setting, and it also makes backend access easier. localStorage can only be written after frontend JS runs, which leaves a gap in the middle. We did not use HttpOnly because the frontend still needed to read the cookie. That was the tradeoff.
  • Unwrapping response.data.data: the backend response shape was fixed: { stat, code, msg, data, traceid }. The interceptor always returned response.data.data, so business code worked directly with the business object. Combined with Swagger-generated types, the business layer ended up feeling almost like local function calls.
  • isDebug=1 in development: this was a debug header we agreed on with the backend. When present, the backend skipped part of the login check, which made local integration much smoother. Production never sent it, so there was no security issue there.
  • The redirect_url on 401: we used encodeURIComponent(window.location.href) to pass the current URL to SSO. After the user signed back in, SSO would send them back. URL query state, like filters, could survive, but in-memory form drafts would still be lost. That was an obvious place for future improvement through localStorage drafts.
  • Why we did not reject on business-level code=401: that was a backend convention. Their code=401 meant “not logged in, but return null quietly and let the business decide whether to show a message.” That is different from a network-layer HTTP 401, which the outer error handler treated as “jump back to SSO.”
How I would say this in an interview

“We were using the company’s SSO, and several subsystems shared one login state. On the frontend we had to solve four things: get login state, inject the token, handle business errors consistently, and recover when the token expires. If every call site touches that logic, the codebase turns into hundreds of repeated lines very quickly.”

“So I pulled all auth and error handling into the two interceptors on our axios instance: the request interceptor reads the token from cookie and injects it into the Authorization header; the response interceptor unwraps data.data so business code stays clean, then handles stat and code in one place — business failures show a toast, and HTTP 401 clears the token, jumps back through SSO, and carries a redirect_url so the user returns to the same page.”

“That lets business code read almost like a local function call: const user = await getUserInfo(). It gets a typed UserInfo directly, with no auth branching and no repeated error boilerplate. Combined with Swagger-generated types, the whole chain stays type-safe.”

What if a user is on a long form, the token expires, and they lose their form state after re-login?
That really was a pain point we discussed at the time. Right now, URL query state survives because redirect_url carries the full URL back, but in-memory form state still gets lost. The next improvement would be localStorage drafts for long forms. We already had drafts in the workspace / page-builder module, but not yet in every other place. If an interviewer asks, I would answer that honestly rather than pretend it was fully solved.
What if the business wants to handle an error quietly instead of letting the interceptor show a toast?
That is an edge case. By default, the interceptor shows the toast and rejects the Promise, and business code only needs to handle it in .catch. For the smaller number of scenarios where a toast is not desirable, like “the user really does not need to see this error,” a custom request config flag like silent: true can tell the interceptor not to show the toast. That follows the same principle as elsewhere in the project: simple by default, but switchable for the smaller set of special cases.
Why not use axios config.validateStatus for all non-2xx responses?
validateStatus only works at the HTTP status level. Our business success rule was response.data.stat === 1. That means the HTTP response can be 200, but the business result is still a failure. So we still had to handle the business layer in the response interceptor. Network errors, like HTTP 4xx/5xx, were handled by axios’s error flow. The two layers stayed cleanly separated.
How did the Swagger-generated types line up with your response-unwrapping logic?
The Swagger generator knew the backend returned { stat, code, data, ... }, and the generated API function types matched the inner data.data, not the outer response shell. That meant a call like const user = await getUserInfo() gave the business layer a UserInfo directly, which matched the interceptor’s unwrap behavior exactly.

04Complex nested forms

Scenarios like purchase-limit rules and course relationships involved deep nesting plus heavy field linkage.

Problem setup

A few kinds of forms in this admin were much more complex than a normal CRUD page:

  • Purchase-limit rules: conditions combine user type, grade, historical order count, purchase time, and more, often with AND/OR relationships.
  • Course relationships: mutual exclusion, substitution, and upgrade relationships between courses, and one course can belong to several relationship chains.
  • Channel configuration: one channel can carry several customer-service accounts, and each account has its own service and attribution rules.

If we had built those directly with raw AntD Form, we would have ended up with:

  • Field linkage implemented through useEffect + form.getFieldValue + form.setFieldValue, which means too many side effects and poor traceability.
  • Dynamic add/remove logic for nested items, like adding and removing conditions inside one rule, becoming manual state-management code that is easy to break.
  • The form structure getting tightly coupled to the component implementation, so moving the rule schema to backend configuration later would mean code changes again.

Core idea

Use Formily’s “reaction model” to turn field linkage from imperative useEffect code into declarative reactions. The schema describes things like: when field A changes, how should field B’s visibility, default value, or options change? All linkage rules stay in the schema instead of being scattered through the business component. Going one step further, the schema itself can be fetched from the backend, which makes the rules configurable.

Code comparison
Before · raw AntD Form + useEffect for linkage
const RuleFormBefore = () => {
  const [form] = Form.useForm()
  const [subOptions, setSubOptions] = useState([])

  // 监听 type 变化，更新 subType 的选项
  useEffect(() => {
    const type = form.getFieldValue('type')
    if (!type) return
    fetchSubOptions(type).then((opts) => {
      setSubOptions(opts)
      form.setFieldValue('subType', undefined) // 清空 subType
    })
  }, [form.getFieldValue('type')])

  // 监听 subType 变化，控制 detail 字段是否显示
  const subType = Form.useWatch('subType', form)
  const showDetail = subType === 'advanced'

  return (
    <Form form={form}>
      <Item name="type"><Select options={TYPE_OPTIONS} /></Item>
      <Item name="subType"><Select options={subOptions} /></Item>
      {showDetail && <Item name="detail"><Input /></Item>}
    </Form>
  )
}
// The linkage logic is scattered through the component, which makes reuse, testing, and configuration much harder.
After · use Formily reactions to describe linkage declaratively
const schema = {
  type: 'object',
  properties: {
    type: {
      type: 'string',
      title: '类型',
      'x-component': 'Select',
      enum: TYPE_OPTIONS,
    },
    subType: {
      type: 'string',
      title: '子类型',
      'x-component': 'Select',
      // 联动：依赖 type，type 变化时异步拉选项 + 清空自己
      'x-reactions': {
        dependencies: ['type'],
        fulfill: {
          run: '$self.setState({ loading: true })',
          state: {
            dataSource: '{{$deps[0] ? fetchSubOptions($deps[0]) : []}}',
            value: '{{$deps[0] !== $self.modified ? undefined : $self.value}}',
          },
        },
      },
    },
    detail: {
      type: 'string',
      title: '详情',
      'x-component': 'Input',
      // 联动：依赖 subType，subType=advanced 时才显示
      'x-reactions': {
        dependencies: ['subType'],
        fulfill: {
          state: { visible: '{{$deps[0] === "advanced"}}' },
        },
      },
    },
  },
}

const RuleFormAfter = () => (
  <FormProvider form={form}>
    <SchemaField schema={schema} />
  </FormProvider>
)
// All linkage rules are now concentrated in the schema, and the whole schema could even move to backend configuration.
Key details
  • What reactions really are: an observer model between fields. dependencies says “whose changes do I care about?” and fulfill says “what should I do when that change happens?” Formily keeps an internal dependency graph, so only fields that genuinely depend on the change get recalculated, which is good for performance.
  • Custom field registration: we wrapped project-specific fields like Cascader and MultipleSelect under components/formily/, then registered them through SchemaField.register. After that, a schema could use them directly with something like 'x-component': 'Cascader'.
  • Dynamic schema: the schema itself is just JSON, so it can be fetched from an API. For example, purchase-limit schemas could be delivered from a backend configuration center, and the frontend could render them directly. That means new rules can go live without code changes.
  • Validation: Formily already supports common validators like required, format, max, min, and pattern, and it also supports custom validator functions. We registered shared project validators under utils/validator.ts, things like mobile number validation, ID validation, and duplicate checks.
  • Performance: Formily updates at a fine-grained level, so changing one field only re-renders fields that actually depend on it. Our biggest forms reached 60+ fields and still felt smooth.
How I would say this in an interview

“This admin had a few kinds of forms that were much more complex than normal CRUD forms: purchase-limit rules, course relationships, channel configuration. They had deep nesting and a lot of field linkage. If we had built that with raw AntD Form, we would have ended up with a pile of useEffect, getFieldValue, and setFieldValue, with linkage logic scattered all over the business component.”

“So I chose Formily’s reaction model. Every linkage rule lives in schema — who depends on whom, and what changes when that dependency changes. The whole schema is plain JSON, which even leaves room to move it into backend configuration later, so new rules can go live without a code change.”

“We also built a small library of project-specific fields under components/formily/: Cascader, multi-select, step-number input, trim input, and so on. So when business code writes a schema, it can just say 'x-component': 'Cascader'. Together with Formily’s validation system, that gave us large 60+ field forms that still felt smooth, readable, and testable.”

Formily has a real learning curve. How did you get the team comfortable with it?
Honestly, there was a learning curve. A new teammate usually needed about a week to build their first complex Formily form. I did three things: 1. wrote down the high-frequency Formily patterns from this project as internal notes; 2. paired with a few teammates early on and used PR review to explain reactions and lifecycle behavior; 3. whenever a tricky linkage case came up, I took it first and then wrote the solution back into the docs. After roughly two weeks, the team was in a much more comfortable place.
Why not build your own schema-based form system?
We did evaluate that. The upside of building our own would have been “perfect fit for the business,” but the downside was obvious: we would have had to build the reaction model, validation, dynamic add/remove, and other base infrastructure ourselves. Formily was already mature in those areas, and we did not have a business need unique enough to justify building our own wheel. In B-side work, it is important to stay disciplined about not building unnecessary infrastructure.
What is the real difference between Formily and raw AntD Form?
AntD Form is imperative: you call getFieldValue, setFieldValue, and usually glue linkage together with useEffect. Formily is declarative plus reactive: the schema declares dependencies between fields, and Formily internally manages the dependency graph so changes propagate automatically. The biggest practical difference is where the linkage logic lives: with raw AntD Form, it is scattered through the business component; with Formily, it is concentrated in the schema.
How are the expression strings inside x-reactions evaluated? Is there a security risk?
Formily uses its own expression engine, not raw eval. Those expressions can access scoped values like $self, $deps, and $form, and we can also register our own custom scope helpers. Because the evaluation is controlled, it does not open the door to arbitrary code execution.
How did you guarantee performance on large forms? Didn’t 60+ fields feel slow?
Formily uses fine-grained reactivity: only fields that actually depend on the changed value re-render, instead of re-rendering the whole form. Our biggest forms had 60+ fields, and we could not feel input lag. If we ever had to go much further, into hundreds of fields, we could still optimize with things like <FormConsumer> for partial subscriptions or virtualized rendering inside <ArrayItems>.

05Multi-environment builds

Four environments: dev, mock, gray, and prod. They had to stay cleanly separated.

Problem setup

We had four environments: dev (development), mock (mock data), gray (staging / gray release), and prod (production). Each one had its own backend domain, SSO address, and log-reporting address. We could not let someone accidentally point local startup at production and write dirty data into the real environment, and we also could not let a gray build accidentally talk to the dev backend.

Core idea

Explicit injection, centralized declaration, and one runtime consumption point. No vague defaults, and no overloading something fuzzy like NODE_ENV to mean too many things. We explicitly injected mode=xxx through cross-env, then defined all environment differences centrally through hostEnum in src/config/index.ts, and let business code read only from config.

Code example
// package.json scripts
{
  "dev": "cross-env mode=dev max dev",
  "dev-mock": "cross-env mode=mock max dev",
  "gray": "cross-env mode=gray max dev",
  "build": "cross-env mode=production max build",
  "build-dev": "cross-env mode=dev max build",
  "build-gray": "cross-env mode=gray max build",
}

// src/config/index.ts
export const hostEnum = {
  mock:       'http://localhost:3001/mock',
  dev:        'https://api-dev.xxx.com',
  gray:       'https://api-gray.xxx.com',
  production: 'https://api.xxx.com',
}

export default {
  host: hostEnum[process.env.mode as keyof typeof hostEnum] ?? hostEnum.dev,
}

// .umirc.ts — the development proxy uses the same hostEnum
const proxy = {
  '/proxyApi': {
    target: process.env.mode ? hostEnum[process.env.mode as keyof typeof hostEnum] : '',
    changeOrigin: true,
    rewrite: (path) => path.replace(/^\/proxyApi/, ''),
  },
}
Key details
  • cross-env made env injection cross-platform: Windows, Mac, and Linux all handle environment-variable syntax differently, and the team had Windows users. That made cross-env genuinely important here.
  • CI scripts were self-explanatory at a glance: if the release script said npm run build-gray or npm run build, everyone could tell immediately which environment the build belonged to. There was no “I’m not actually sure which environment this artifact is for.”
  • Business code did not need to understand environment details: need the host? read import config from '@/config'. Need to check whether it is dev? read process.env.NODE_ENV. No random if/else branches all over the project.
How was the mock environment implemented?
We used Umi’s built-in mock capability. We put mock handlers under the mock/ directory, and when npm run dev-mock started up, Umi intercepted matching requests and returned mock data. The upside was that it integrated naturally with Umi’s route system and needed almost no extra setup. We did not bring in something like MSW — MSW is more useful when you are building a frontend component library in isolation, and for a large internal admin we did not need that extra layer.
If something breaks after a gray release, how do you locate it quickly?
Gray-environment requests carried an X-Env: gray header, so backend logs could distinguish them. Frontend log reporting also carried the mode field. When something went wrong, we used the troubleshooting tool, the check-tool module, entered a user ID, pulled the recent request chain, and filtered it by environment. That was much faster than trying to inspect production logs directly.
