关键链路 1：启动与权限

用户访问页面时，前端的完整启动流程：

SSO 登录 → token 写入 Cookie → getInitialState() → 拉取菜单 + 功能权限 → access 配置生成 → 路由守卫 + 按钮级鉴权

Umi Max 的 initialState 做一次性权限拉取，access.ts 根据菜单列表 生成几十个布尔权限开关（如 canSeePresentCoupon），业务代码通过 <Access accessible={access.canSeeXxx}> 或 useAccess() hook 消费，一处配置、全站生效。

关键链路 2：请求与鉴权

业务调用 → axios 实例 → 请求拦截器注入 token → 后端 → 响应拦截器 → stat/code 校验 → 统一 message 提示

响应拦截器里把 401 自动登出 + 跳 SSO 和 业务错误统一弹 toast 两件事做掉，业务代码里只需要关心 happy path。错误码约定在拦截器里收敛，不扩散到业务层。

关键链路 3：路由与菜单

routes/index.ts 集中声明 → 每个节点带 access 字段 → Umi 生成路由表 → ProLayout 自动渲染侧边菜单 → 无权限节点自动隐藏

路由表由后端权限数据驱动 —— 一份路由声明，路由跳转、菜单渲染、权限拦截三件事同时搞定。 新增业务模块只需要加一段 route + 对应 access key，不用手动改菜单。

目录结构概览

src/
├── apis/                 Swagger 自动生成 + 手写 API 层
├── components/           通用组件（Formily 扩展、上传、筛选、排序、机型预览…）
├── config/               环境配置（host、mode）
├── constants/            枚举、常量
├── hooks/                自定义 hooks（权限校验、下拉枚举、列表搜索…）
├── layouts/              顶层布局
├── pages/
│   ├── template/         项目中心
│   ├── present/          礼品中心
│   ├── tools/            小工具
│   ├── user-portrait/    用户画像
│   └── work-space/       工作台（页面搭建器）
├── routes/index.ts       全局路由声明（含 access）
├── store/                Redux Toolkit: config / user / checkTool
├── utils/                axios / 登录 / 校验 / 日志 / 日期
├── access.ts             权限配置
└── app.tsx               Umi 运行时钩子（rootContainer / layout / getInitialState）
