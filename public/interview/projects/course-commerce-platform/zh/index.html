项目深度梳理

课程商业平台

一个支撑课程业务线的 B 端运营管理后台 —— 5 大业务域（项目 / 礼品 / 工具 / 画像 / 工作台）、 20+ 二级菜单统一在一个系统里，由一套基于 Umi Max 的前端架构 + 细粒度权限体系撑起来。

01概览

一句话说清项目

一个面向内部运营团队的商业化中台，把课程业务下所有和"赚钱" 相关的运营操作收敛到同一个后台里：从商品定义（课程、盒子、优惠券）、营销工具（推广、渠道码、 定制活动）、用户规则（限购、画像），到落地页的搭建与发布，形成一条完整的运营闭环。

业务域一览

项目中心
课程项目 / 导航项目 / 模板中心 / 组件库 / 小程序，落地页搭建的入口。
礼品中心
课程商品（供应链 / 商品中心）、实物商品（课程盒子 / 实物盒子）、虚拟商品（优惠券）。
小工具
推广工具、定制活动、渠道活码、排查工具。营销运营的日常家伙事儿。
用户画像
用户限购、课程限购、课程关系，控制哪些用户能买哪些课。
工作台
落地页搭建器（7 步向导 + Formily 表单 + 真机预览）。作为一个入口存在。
背景 (Situation)
课程业务的运营链路散落在多个老后台里 —— 商品在一个系统，优惠券在一个系统，落地页要让前端 手写，限购规则靠后端改代码。运营每做一次活动都要跨 3-4 个系统协同，前端也疲于给各系统打补丁。
目标 (Task)
把这些散落的运营能力收敛进一个统一的后台，提供一致的交互、一致的权限、一致的数据口径， 并让非技术的运营同学能端到端自助完成大部分日常操作。
行动 (Action)
基于 Umi Max 搭建前端底座，约定式路由 + access 插件做细粒度权限；业务分层为 "业务模块 / 平台能力 / 基础设施"三层；用 Redux Toolkit 管理全局状态，Formily 做复杂表单，ahooks 统一异步与交互；@dandelion/swagger 自动生成 API，收敛前后端契约。
结果 (Result)
一个后台覆盖 5 大业务域、20+ 二级页面；运营跨系统切换的成本大幅降低； 页面搭建从前端排期变成运营自助，前端团队从"做页面"转向"把平台做好"。

注：本项目中也包含一个低代码页面搭建器（工作台模块），但由于团队另有专门的低代码平台作为亮点项目， 这里只做概述、不做深挖。更多细节见「重点模块」页。
