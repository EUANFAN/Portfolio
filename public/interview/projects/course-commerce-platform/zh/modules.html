05重点模块

5 个重点模块，逐一拆解

这里不是按业务域讲，而是按技术能力 / 工程抽象讲 —— 项目里真正有复用度、有沉淀、也最能经受追问的 5 个模块： 权限、CRUD 开发框架、复杂表单、OSS 上传、规则引擎前端。 低代码工作台作为其中一个业务模块存在，文末一笔带过。

01 · 权限模块 src/access.ts · src/hooks/useAuthCheck.ts

后端权限列表直接驱动前端的路由 + 菜单 + 按钮。 菜单权限和功能权限明确分层，边界清楚不串线。

数据源
Umi Max 的 getInitialState() 里一次性拉 { menus, permissions }，按 type=0/1（菜单）和其他（按钮）拆成两棵树。
路由 + 菜单
access.ts 生成几十个布尔开关（如 canSeePresentCoupon），<Access> 做路由守卫，ProLayout 自动按 access 过滤菜单，无权限节点自动隐藏。
按钮级
useAuthCheck('project-center-project-list-course-edit') 返回 boolean，控制按钮 / Tab / 表格操作列的可见性。
兜底
手动 URL 访问越权节点时走 403 页兜底；岗位变更时 initialState 可主动刷新。
Umi Max access initialState ProLayout 50+ 权限 key 10+ 岗位隔离
02 · CRUD 开发框架 src/hooks/useTableSearch · useTableSet

20+ 列表页共享的底层能力。写一个新列表页只需要声明 columns + searchForm + fetchFn，搜索 / 分页 / 列持久化 / URL 同步全部由 Hook 接管。

useTableSearch
返回 { search, reset, formValue }。搜索时自动 reset current，翻页时保留筛选，重置时清 params，并把搜索条件同步到 URL —— 刷新页面状态不丢。
useTableSet
接管自定义列：列宽、列显隐、操作列渲染。返回 { width, render }，用户调整的列配置走 localStorage 持久化，换人换浏览器不掉。
配合方式
和 ahooks 的 useAntdTable 组合 —— 数据源 useAntdTable 管、搜索分页 useTableSearch 管、列配置 useTableSet 管，三者职责互不重叠。
实际效果
新列表页从 500+ 行压到 ~150 行，20+ 页面交互行为开箱一致，业务开发不用再"想一次默认行为"。
ahooks.useAntdTable ProTable URL 同步 localStorage 持久化 20+ 列表页复用
03 · 复杂表单模块 src/components/formily/

优惠券、限购规则、商品这类深层嵌套 + 字段联动 + 大量校验的场景，用 Formily 把状态管理让渡给框架，业务代码只关心数据结构。

声明式 schema
嵌套结构、字段依赖、条件显隐、校验规则全部在 schema 里声明，避免手写 onChange + 一堆 if 维护表单状态。
Effects 做联动
用 onFieldValueChange 监听字段、setFieldState 改下游状态，复杂联动（如优惠券类型切换触发 10+ 字段显隐）用 20 行代码搞定。
自研扩展表单项
级联选择、多选、Trim 输入、数字输入、开关、颜色选择器（react-colorful）、代码编辑器（@uiw/react-codemirror）、标签选择器等 15+ 个 Formily 适配组件。
下拉枚举统一
useSelectEnum() 统一所有下拉选项数据源，枚举更新一处、全站同步。
Formily 2 onFieldValueChange / setFieldState react-colorful @uiw/react-codemirror 15+ 扩展表单项
04 · OSS 上传模块 src/components/upload-component/ · src/hooks/useOssData.ts

图片 / 视频 / 文件 / Excel 导入全部走同一套签名直传方案，前端直接 POST 到 OSS，不经过业务服务器中转。

统一入口
<ImgUpload> / <VideoUpload> / <Dragger> / Formily 适配版，统一封装签名、校验、进度、回显逻辑。
签名直传
getOssToken() 拿 { OSSAccessKeyId, signature, policy, host, dir } → FormData 带上 key / policy / signature 直接 POST 到 OSS host → 返回 fileUrl 后拼成 OSS_CDN_NAME/OSS_PUBLIC_PATH/fileUrl 写回字段。
视频处理
封面直接用 ?x-oss-process=video/snapshot,t_0,f_jpg,w_110,h_110 生成，无需后端二次转码。
CSV 导入校验
文件导入场景上传后后端返回 { result, list }，前端直接表格回显校验结果；校验失败时通过 isValidate 字段联动阻塞表单提交。
OSS 签名直传 前端零转发 x-oss-process CSV 校验表格 Formily 适配
05 · 规则引擎前端 src/pages/user-portrait/ · src/components/rules/

把"哪些用户能买哪些课"从后端硬编码变成前端可配置的规则树。 用户限购 / 课程限购 / 课程关系三个正交视角，合起来就是一套完整的营销准入策略。

用户限购
按用户标签、身份、消费历史等维度组合规则。规则语义：「满足某组条件 → 命中限购」。
课程限购
从课程维度反过来配：这门课谁能买、每人几份、是否有阶梯定价、是否有老带新折扣。
课程关系
互斥 / 替代 / 升级三种关系：买了高级版能否买入门版、买了 A 是否就不能买 B，下单校验时后端读这张关系表。
规则组合器 UI
Formily 动态 schema 递归渲染规则树，AND / OR 嵌套 + 条件项。前端吐 JSON DSL，后端解析执行 —— 新增一种规则类型只扩 schema，不改 UI。
Formily 动态 Schema JSON DSL 条件组合器 UI 规则引擎前端

补充：低代码工作台

src/pages/work-space/ 是整个后台中的一个业务模块 —— 7 步向导 + Formily 表单 + 手机端实时预览 + 保存 / 预览 / 上线闭环，支持 8 种落地页模板和多端投放（H5 / 小程序）。 因为低代码能力在团队的另一个亮点项目 Low-Code Campaign Platform 里深度和完整性更高，这里不再展开。

模块协同上的几个关键设计

  • 一套权限模型覆盖全站 —— 50+ 权限 key 同时管住菜单、路由、按钮，新模块加一个 key 就接入全站。
  • 通用组件跨模块一致 —— 表单搜索组、列表操作下拉、上传、Formily 扩展表单项在所有模块里是同一个样子、同一种行为。
  • API 契约自动同步 —— 后端改接口 → Swagger 拉一次 → 前端编译就能发现不兼容的地方，不用靠联调。
  • 共享的 Redux 切片 —— config（全局配置）/ user（用户信息）/ checkTool（排查工具状态），跨页面共享不层层透传。
